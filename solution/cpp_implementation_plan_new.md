# C++ 实现方案拆解（布局求解题）- 当前代码对齐版

## 1. 问题定位

这道题本质上是一个带约束的二维矩形布局问题：

- 将多个矩形物体摆放到给定房间轮廓内
- 满足不越界、不重叠、不遮挡门禁区域
- 物体仅允许 0 度和 90 度旋转
- 冰箱需要额外考虑开门保留区
- 在可行前提下优先贴墙摆放

当前代码采用的核心思路不是在连续平面直接求最优，而是先做几何建模，再生成候选位置，最后通过 DFS / 回溯搜索一组可行解。这种做法结构清晰、易调试，也方便继续迭代。

---

## 2. 总体实现策略

当前工程可以概括为四层：

1. 输入解析层
   - 支持文本格式与 JSON 格式
   - 自动根据文件后缀走不同解析逻辑

2. 几何建模层
   - 提供点、线段、矩形、多边形等基础结构
   - 负责轮廓内判断、矩形相交判断、门区判断等

3. 候选位置生成层
   - 第一阶段：优先沿真实轮廓边生成贴墙候选
   - 第二阶段：再补充内部候选作为兜底

4. 搜索求解层
   - 先对物体排序
   - 再用 DFS / 回溯搜索一组合法摆放

这四层对应当前代码中的 `parser.*`、`geometry.*`、`solver.*` 和 `main.cpp`，职责划分比较明确。

---

## 3. 当前版本建模假设（与代码一致）

为了控制实现复杂度，当前代码采用以下假设：

- 房间轮廓按正交多边形处理（边平行于 x 轴或 y 轴）
- 门为水平或竖直门
- 所有物体均为轴对齐矩形
- 物体只允许 0 度和 90 度旋转
- 贴墙候选只沿真实的水平边或竖直边生成
- 若轮廓出现斜边，当前贴墙逻辑会跳过该边
- 家具之间当前实现不允许边接触
- 家具与门区当前实现也不允许边接触

其中“边接触也算冲突”是当前代码的重要实现细节：虽然几何函数 `IsRectOverlap` 支持“允许接触”的模式，但在实际摆放校验中调用的是 `IsRectOverlap(..., false)`，所以当前求解器使用的是更严格的约束。

## 5. 核心数据结构设计

### 5.1 基础几何结构

当前代码使用以下基础结构：

- `Point`：二维点 `(x, y)`
- `Segment`：线段，由两个点构成
- `Rect`：轴对齐矩形，用 `(x1, y1, x2, y2)` 表示

因为所有物体只支持 0 / 90 度旋转，所以使用 AABB（轴对齐包围盒）表示矩形是合适的：

- 表达简单
- 旋转后仍易处理
- 相交检测成本低

### 5.2 题目对象结构

当前代码中的业务对象包括：

- `Door`：门线段 + 门类型（`INWARD / OUTWARD`）
- `Item`：物体名称、类型、长、宽
- `Placement`：最终摆放结果
  - `item_index`
  - `center`
  - `angle`
  - `body`
  - `forbidden_zone`（冰箱专用）
- `Problem`：输入整体
- `Solution`：输出整体

此外，`Solver` 内部还定义了 `Candidate`，用于表示候选位置：

- 中心点
- 角度
- 本体矩形
- 是否为贴墙候选（`wall_hugging`）

## 6. 几何模块职责

几何模块是整个求解器的底座。

### 6.1 矩形工具函数

当前代码中已具备：

- 根据中心点和宽高生成矩形：`MakeRectFromCenter`
- 计算包围盒：`ComputeBoundingBox`
- 矩形归一化：`NormalizeRect`
- 计算矩形宽高：`RectWidth / RectHeight`
- 获取矩形四角：`RectCorners`
- 判断矩形是否重叠：`IsRectOverlap`

需要注意：

- `IsRectOverlap(a, b, true)`：边接触也算冲突
- `IsRectOverlap(a, b, false)`：只有面积相交才算冲突

而当前求解流程在门区冲突、家具冲突、禁区冲突检查时传入的都是 `false`，因此当前实现效果是：

- 边接触也不允许
- 必须留出明确缝隙

### 6.2 点与多边形关系

当前代码使用射线法判断点是否在多边形内：

- `IsPointOnSegment`：判断点是否在线段上
- `IsPointInPolygon`：判断点是否在多边形内

当前实现把“点在边上”也视为在内部，这一点对贴墙候选非常重要，因为贴墙时矩形边往往恰好落在轮廓边界上。

### 6.3 矩形是否在轮廓内

`IsRectInsidePolygon` 是核心合法性判断函数，当前代码思路是：

1. 检查矩形四个角是否都在轮廓内或边界上
2. 检查矩形边与轮廓边是否发生真正穿出轮廓的相交

这样可以较稳定地过滤掉越界候选。

## 7. 门约束的实现方式

当前代码会先把门转换成一个禁放矩形 `door_block_`，后续所有候选都要避开这个区域。

### 7.1 当前实现逻辑

在 `Solver` 构造时：

- 先用 `ComputeBoundingBox(problem_.contour)` 得到 `room_bbox_`
- 再调用 `BuildDoorBlock()` 生成 `door_block_`

### 7.2 内开门

若门为 `INWARD`：

- 先计算门宽 `doorWidth`
- 再将门所在边朝室内方向扩展 `doorWidth` 的深度
- 形成一个近似门扇摆动区域

### 7.3 外开门

当前代码中，`OUTWARD` 不是完全不占室内空间，而是：

- 仍保留一个 `depth = 1` 的近似缓冲带

这是一种偏保守的近似实现，优点是简单；若后续要更贴业务，可以再调整为 0 或改成仅保留门洞通道。

### 7.4 门朝室内方向如何判断

当前代码主要使用：

- 门是水平边还是垂直边
- 配合房间包围盒中心 `room_bbox_` 的相对位置

来判断门区向哪一侧扩展。这种方式实现简单，但本质上仍是近似判断。

## 8. 特殊物体规则

### 8.1 冰箱（FRIDGE）

冰箱除了本体矩形，还要生成一个额外的开门保留区（禁区），该禁区会写入 `Placement.forbidden_zone`。

当前规则可理解为：

- `angle = 0`：在本体上侧额外扩出一条禁区
- `angle = 90`：在本体右侧额外扩出一条禁区

该禁区的作用是：

- 其他物体不能进入这块区域
- 禁区本身也必须在房间轮廓内

### 8.2 其他物体

`SHELF`、`OVERSHELF`、`ICEMAKER` 当前按普通矩形处理：

- 只检查本体是否越界
- 是否碰门
- 是否与其他物体或禁区冲突

## 9. 候选位置生成策略

候选生成是当前代码最核心的部分之一。

整体思路是：

- 不在连续平面暴力搜索
- 而是先生成一批离散的候选位置
- 再在这些候选上做 DFS 组合搜索

### 9.1 第一阶段：真实轮廓贴墙候选

当前代码中的 `GenerateWallCandidates()` 已经不再按房间包围盒四边简单取点，而是：

1. 对每个物体枚举两种角度：0 / 90
2. 用 `RotatedSpanX / RotatedSpanY` 计算旋转后的宽高
3. 遍历 `problem_.contour` 的每一条边
4. 只处理水平边和垂直边
5. 通过 `InferInteriorNormal()` 判断哪一侧是室内方向
6. 沿该边的可放置区间生成一组中心点采样
7. 每个采样点构造候选矩形，并调用 `BasicCandidateValid()` 过滤

因此，当前贴墙候选已经是“沿真实轮廓边”生成，而不是早期版本那种“只贴包围盒四边”。

### 9.2 贴墙采样：自适应步长 + 区间采样

当前贴墙采样不再只取固定三点，而是：

- 用 `ComputeAdaptiveStep(item, angle, true)` 计算贴墙采样步长
- 再通过 `AppendLineSamples()` 在边的有效区间内生成采样点

当前自适应步长规则是：

- 以旋转后较短边为基础
- 贴墙模式取 `base * 0.5`
- 并限制在 `[50, 250]` 的范围内

这样做的好处是：

- 大件物体不会采得过密
- 小件物体也不会过稀
- 候选数量比固定三点更合理

### 9.3 第二阶段：内部候选（兜底）

若第一阶段纯贴墙候选无法求解，当前代码会进入第二阶段，在保留贴墙候选的基础上，为每个物体补充内部候选。

`GenerateInteriorCandidates()` 的逻辑是：

1. 仍以 `room_bbox_` 作为内部扫描窗口
2. 根据物体尺寸和角度确定中心点可取范围
3. 在该范围内做双重循环扫描
4. 每个格点构造候选矩形
5. 用 `BasicCandidateValid()` 做越界与门区过滤
6. 用 `AddCandidateUnique()` 去重

因此，当前内部候选依然是“基于包围盒窗口的离散网格扫描”，只是已经不再使用固定死步长。

### 9.4 内部扫描：自适应步长

当前第二阶段在 `Solve()` 中调用的是：

- `GenerateInteriorCandidates(i, 0.0)`

传入 `0.0` 的含义是：

- 让函数内部自动调用 `ComputeAdaptiveStep(item, angle, false)`

当前内部采样规则是：

- 以旋转后较短边为基础
- 内部扫描取 `base * 0.35`
- 并限制在 `[50, 250]`

因此，当前代码已经实现了“自适应步长内部搜索”，而不是旧版的固定 `1.0` 或固定 `100.0` 扫描。

## 10. 候选合法性与摆放合法性检查

当前代码把合法性分成两层。

### 10.1 `BasicCandidateValid()`

用于候选生成阶段的快速过滤，检查：

- 候选本体矩形是否在轮廓内
- 是否与门区冲突

只有通过这一层，候选才会进入候选池。

### 10.2 `IsPlacementValid()`

用于 DFS 摆放阶段的完整校验，检查：

- 当前本体矩形在轮廓内
- 当前本体不与门区冲突
- 若有冰箱禁区，则禁区本身也必须在轮廓内
- 不与已放物体本体冲突
- 不进入已放冰箱的禁区
- 若当前也是冰箱，则自己的禁区不能压到已放物体本体

### 10.3 当前冲突规则（很重要）

当前代码里，门区冲突、物体冲突、禁区冲突检查统一都使用：

- `IsRectOverlap(..., false)`

这意味着当前版本是：

- 边接触也视为冲突
- 必须留缝

因此，当前求解器比“允许贴边摆放”的版本更保守，也更容易出现候选被过滤的情况。

## 11. 搜索求解层（DFS / 回溯）

当前代码的搜索主线是：

- `TrySolveWithCandidates()`
- `BuildSearchOrder()`
- `DFS()`

### 11.1 搜索顺序

当前 `BuildSearchOrder()` 的优先级是：

1. 冰箱优先
2. 候选数量少的优先（MRV 思路）
3. 面积大的优先
4. 下标小的优先（作为最终兜底）

这个顺序能有效减少回溯分支。

### 11.2 DFS 逻辑

当前 `DFS()` 的流程很标准：

1. 取当前深度要摆放的物体
2. 枚举它的所有候选
3. 将候选转成 `Placement`
4. 用 `IsPlacementValid()` 校验
5. 合法则压入 `placed`
6. 递归摆下一个物体
7. 若失败则回溯 `pop_back()`
8. 全部摆完则写入 `Solution` 并返回成功

当前版本找到第一组可行解就停止，不做全局最优搜索。

## 12. “优先贴墙”在当前代码中的体现

当前代码用的是很清晰的两阶段策略：

### 阶段一：仅贴墙候选

- 为每个物体只生成贴墙候选
- 调用 `TrySolveWithCandidates()`
- 若成功，直接返回

### 阶段二：贴墙 + 内部候选

- 若第一阶段失败
- 则重新生成贴墙候选
- 再补充内部候选
- 去重后再次执行 DFS

这正好体现了“优先贴墙”是偏好而非硬约束：

- 能贴墙解就优先返回
- 贴墙无解再放宽到内部搜索

## 13. 输入输出与交互方式

### 13.1 输入格式

当前代码支持两类输入：

1. 文本格式
2. JSON 格式

并且由 `ParseProblemFromFile()` 根据文件后缀自动分流。

#### 文本格式

```text
CONTOUR 4
0 0
10 0
10 8
0 8

DOOR
4 0 6 0 INWARD

ITEMS 3
item1 FRIDGE 2 1
item2 SHELF 3 1
item3 ICEMAKER 1 1
```

#### JSON 格式

```json
{
  "boundary": [[0, 0], [10, 0], [10, 8], [0, 8], [0, 0]],
  "door": [[4, 0], [6, 0]],
  "isOpenInward": true,
  "algoToPlace": {
    "fridge": [2, 1],
    "shelf-1": [3, 1],
    "iceMaker": [1, 1]
  }
}
```

其中 JSON 解析还做了两件额外适配：

- 自动删除 `boundary` 首尾重复闭合点
- 根据 `algoToPlace` 的 key 推断 `ItemType`

### 13.2 当前程序入口交互

`main.cpp` 当前不是写死路径，而是：

- 从标准输入读取文件路径
- 再调用 `ParseProblemFromFile()` 解析

这意味着当前运行方式更灵活，适合直接切换不同测试文件。

### 13.3 输出格式

当前输出为：

```text
FEASIBLE YES
fridge center=(5733.442, 30606.794) angle=0
iceMaker center=(6103.442, 29466.794) angle=0
```

若无解则输出：

```text
FEASIBLE NO
```

输出由 `PrintSolution()` 统一完成。

## 16. 下一步最值得继续优化的方向

基于当前代码，后续最值得继续做的优化是：

1. 缩小内部扫描范围
   - 当前内部候选仍以 `room_bbox_` 为扫描窗口
   - 后续可改为更局部的可行区域扫描，减少无效格点

2. 减少对包围盒中心的近似依赖
   - `BuildDoorBlock()` 和 `InferInteriorNormal()` 仍在使用 `room_bbox_` 进行兜底判断
   - 后续可改成更精确的法向量和轮廓方向判断

3. 支持斜边
   - 当前贴墙候选仅支持水平 / 垂直边
   - 若轮廓含斜边，当前逻辑会跳过

4. 候选冲突预计算
   - 先建立候选之间的冲突关系表
   - 减少 DFS 中重复几何判断

5. 增加评分函数
   - 当前找到第一组可行解就停止
   - 后续可在多组可行解中选更整齐、更贴墙或更节省空间的布局

6. 进一步清理特殊物体规则
   - 将不同类型物体的禁区或特殊约束参数化
   - 让扩展更多业务规则更容易

## 17. 总结

结合当前代码，最准确的实现描述是：

- 用统一的数据结构表示房间、门、物体和结果
- 用几何函数负责“在不在轮廓内、是否冲突”的底层判断
- 用“真实轮廓贴墙 + 自适应步长内部扫描”生成离散候选
- 用两阶段策略体现“优先贴墙”
- 用 DFS / 回溯在候选集合里搜索第一组可行解

这套方案的优点是：

- 结构清楚
- 与当前代码一致
- 便于调试
- 便于讲解
- 便于继续迭代
